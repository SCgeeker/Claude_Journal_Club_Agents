# /ultrathink - 深度思考與優雅設計模式

## 🎯 適用場景

**✅ 適合使用**:
- 大型功能開發（>100行代碼）
- 系統架構設計
- 複雜演算法實作
- 需要創新解決方案的問題
- 代碼重構與優化

**❌ 不適合使用**:
- 簡單文件讀寫
- 基本查詢操作
- 小型 bug 修復（<10行改動）
- 文檔更新
- 格式調整

---

## 🚀 核心原則（Steve Jobs 哲學）

> "We're not here to write code. We're here to make a dent in the universe."

### 1. Think Different - 質疑假設
- 為什麼一定要這樣做？
- 如果從零開始，最優雅的方案是什麼？
- 有沒有完全不同的角度？

### 2. Obsess Over Details - 細節執著
- 深入閱讀 CLAUDE.md、README.md
- 理解代碼庫的設計哲學
- 學習現有模式和慣例
- 保持一致性

### 3. Plan Like Da Vinci - 大師級規劃
- 先在腦中構建完整架構
- 使用 TodoWrite 創建詳細計畫
- 文檔化設計決策
- 讓計畫本身就是藝術品

### 4. Craft, Don't Code - 工藝而非編碼
- 每個函數名都應該「唱歌」
- 每個抽象都應該感覺自然
- 優雅地處理所有邊界情況
- TDD 是對卓越的承諾

### 5. Iterate Relentlessly - 無情迭代
- 第一版永遠不夠好
- 運行測試、比較結果
- 持續優化直到「insanely great」

### 6. Simplify Ruthlessly - 無情簡化
- 移除一切不必要的複雜性
- "完美不是無可增，而是無可減"

---

## 🛠️ 工作流程

### Phase 1: 深度理解（Think）

```markdown
1. 閱讀相關文檔和代碼
2. 理解問題的本質
3. 質疑所有假設
4. 探索多種可能性
```

### Phase 2: 優雅規劃（Plan）

```markdown
1. 使用 TodoWrite 創建任務清單
2. 設計架構和數據流
3. 識別關鍵抽象
4. 文檔化設計決策
```

### Phase 3: 精湛實作（Craft）

```markdown
1. 先寫測試（TDD）
2. 實作核心功能
3. 優雅處理邊界情況
4. 重構直到完美
```

### Phase 4: 嚴格驗證（Verify）

```markdown
1. 運行所有測試
2. 檢查代碼品質
3. 驗證性能
4. 確保文檔完整
```

## 📊 成功指標

任務完成時，應達到以下標準：

- ✅ **優雅性**: 代碼讀起來像散文
- ✅ **簡潔性**: 沒有冗餘或複雜性
- ✅ **完整性**: 所有邊界情況都被處理
- ✅ **可測試性**: 100% 測試覆蓋率
- ✅ **一致性**: 符合代碼庫風格
- ✅ **文檔性**: 設計決策被記錄

## ⚠️ 重要提醒

### 避免過度設計

即使在 ultrathink 模式，也要：
- ✅ 保持輸出簡潔（除非需要詳細解釋）
- ✅ 先執行工具，再深度思考
- ✅ 分階段實作，不要一次完成所有
- ✅ 實用主義 > 完美主義

### 自動降級

如果任務明顯簡單（<5分鐘完成），自動切換回標準模式並提醒用戶：

```
"這個任務相對簡單，我將使用標準模式快速完成。
如需深度分析，請明確要求。"
```

```

     設計一個優雅的永久筆記生成系統，
```
